# Newsphere Live

A self-updating semantic map of today's news. Every 6 hours, a pipeline fetches articles from 11 RSS feeds, embeds them using a sentence transformer, clusters them by topic, and publishes an interactive 3D visualization to GitHub Pages. No backend, no database, no recurring costs.

**Live site** → [rahulrayy.github.io/newsphere-live](https://rahulrayy.github.io/newsphere-live)

---

## What it does

The map shows hundreds to thousands of news articles arranged in 3D space where proximity reflects semantic similarity. Articles about the same story cluster together regardless of how differently they phrase it. You can spin the map, zoom in, hover over any dot to see the headline and source, click it to open the article, or click a topic in the sidebar to see every article in that cluster. A timeline slider lets you filter by date. New articles added since the previous run are highlighted in green.

A digest page summarises the top topics with clickable headlines. Visitors can subscribe to receive this as a daily email.

---

## How it works

Each article gets converted into a 384-dimensional vector using the `all-MiniLM-L6-v2` sentence transformer. These vectors capture meaning rather than exact words, so "Fed holds rates steady" and "Federal Reserve keeps interest rates unchanged" end up close together in the embedding space even though they share almost no words.

UMAP reduces those 384 dimensions down to 3 so the articles can be plotted in a browser. HDBSCAN then finds dense regions in that 3D space and labels them as topic clusters. Each cluster gets a human-readable label generated by c-TF-IDF, which finds the words most distinctive to that cluster compared to everything else.

Because UMAP is stochastic, the layout would normally flip and rotate unpredictably between runs. The pipeline fixes this using Procrustes alignment. After each run, the new coordinates are aligned to the previous run's using URL-matched articles as anchors. The global orientation stays consistent for repeat visitors even as new articles arrive and old ones age out.

## Background

This project is a standalone successor to [Newsphere](https://rahulrayy.github.io/Corpus-Map/index.html), a static semantic map of 7,000 news articles from 2018 built as a personal project while studying Computational Social Science at Leiden University. That project included quantitative evaluation of cluster quality using silhouette scores and topic coherence metrics across different parameter settings. The architecture of Newsphere Live reflects those findings directly — the HDBSCAN minimum cluster size fraction, cosine similarity dedup threshold, and UMAP neighbourhood parameter were all informed by what worked and what did not in the static version.

The original Newsphere site remains unchanged as a self-contained portfolio piece.

## Pipeline

The pipeline runs as a GitHub Actions workflow on a cron schedule, rebuilding every 6 hours at 00:00, 06:00, 12:00, and 18:00 UTC. Each step is a separate Python script.

```
fetch_articles.py    pulls 11 RSS feeds, deduplicates by normalised title, merges into rolling 7-day store
embed.py             encodes each article (title + description) using all-MiniLM-L6-v2
cluster.py           semantic dedup, UMAP 3D, Procrustes alignment, HDBSCAN, c-TF-IDF labelling
diff.py              compares current output to previous run by URL to find new articles
validate.py          checks point count and cluster count meet minimum thresholds before committing
digest.py            generates digest.html and sends the email broadcast via Resend
```

If validate.py fails, the pipeline exits with a non-zero code and nothing gets committed. The site stays on the last good output rather than publishing a broken JSON.

---

## Deduplication

Cross-publisher coverage is handled in two passes.

The first pass runs during fetch. Titles are lowercased, punctuation is stripped, and whitespace is collapsed. Exact matches after this normalisation are dropped immediately. This catches same-publisher re-posts and minor headline edits.

The second pass runs after embedding. Articles whose embedding vectors have a cosine similarity above 0.90 are collapsed into a single point. The version with the longer description is kept as canonical, and the absorbed outlets are stored in an `also_covered_by` field. This field is surfaced in the UI so you can see at a glance how many outlets covered the same story.

## Zero-cost architecture

Every component was chosen with a hard constraint of no recurring infrastructure costs.

| Component | Service | Cost |
|---|---|---|
| ML pipeline compute | GitHub Actions (public repo, unmetered) | free |
| Static hosting | GitHub Pages | free |
| News data | RSS feeds (no auth required) | free |
| Email delivery | Resend free tier (100 emails/day) | free |
| Subscriber storage | Resend Audiences | free |
| Subscribe endpoint | Cloudflare Workers free tier | free |

The key architectural decision that makes this possible is keeping everything static. The frontend is plain HTML, CSS, and JavaScript loading a JSON file. There is no server, no database query, and no API call at page load time. GitHub Pages serves static files for free indefinitely.

GitHub Actions is free for public repositories with no minute cap. Each pipeline run takes around 6 to 8 minutes, so all four daily runs are unmetered and cost nothing.

The subscribe form is the one part that needs a server-side component, because Resend's API key cannot be exposed in client JavaScript. A small Cloudflare Worker handles this. It receives the email address, adds the contact to the Resend audience using the API key stored as a Worker secret, and returns a success response. The Worker runs on Cloudflare's free tier which allows 100,000 requests per day.

## Running locally

```bash
git clone https://github.com/Rahulrayy/newsphere-live
cd newsphere-live
pip install uv
uv pip install -r pipeline/requirements.txt

python pipeline/fetch_articles.py
python pipeline/embed.py
python pipeline/cluster.py
python pipeline/diff.py
python pipeline/validate.py
python pipeline/digest.py
```

The first run of embed.py downloads the MiniLM model (~90MB) and caches it. Subsequent runs use the cache. The full pipeline takes around 2 to 3 minutes locally on a modern laptop.

Open `index.html` directly in a browser after running the pipeline to view the visualization with today's data.

## Known limitations

**Bootstrap period.** On the first run, the rolling store is empty so the corpus is only whatever the RSS feeds surface at that moment, typically 300 to 450 articles. The map is usable immediately but clusters become noticeably tighter and more coherent after a few days once the store has accumulated enough data.

**UMAP cluster drift.** Procrustes alignment locks the global orientation of the map but cannot prevent individual clusters from shuffling relative to each other when UMAP's internal graph changes between runs. The overall frame is stable but the internal layout is not fully deterministic across long time horizons.

**RSS sampling gap.** Feeds are fetched every 6 hours. High-volume publishers rotate articles faster than this window, so some articles that existed briefly between fetches are missed. More frequent fetching would close this gap but was left for a future iteration.

**Resend free tier.** The free tier supports up to 100 email subscribers. Beyond that a paid plan is required.
